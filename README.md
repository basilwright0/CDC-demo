# Near-real-time CDC denormalization pipeline

A demo data-engineering project: a normalized MySQL OLTP database is captured with
**Debezium** change data capture, streamed through **Kafka**, denormalized in near-real-time
by a **Kafka Streams** application, and written back into a separate MySQL **read model** by a
**Kafka Connect JDBC sink**. Everything runs in Docker for reproducibility.

This is the streaming equivalent of maintaining a materialized view — a CQRS-style read model
kept continuously in sync with its source of truth.

## Architecture

```
data generator                                            Kafka UI (observe)
      │                                                          ▲
      ▼                                                          │
MySQL (source, OLTP)                                        Schema Registry
      │                                                      (Avro schemas)
      ▼
Debezium source connector  ──►  raw CDC topics (one per table, in Kafka)
                                          │
                                          ▼
                              Kafka Streams app  (FK-join + aggregate → denormalize)
                                          │
                                          ▼
                                 denormalized topic
                                          │
                                          ▼
                              JDBC sink connector (upsert by PK)
                                          │
                                          ▼
                              MySQL (sink, read model — separate instance)
```

The source and sink are **separate MySQL instances**. This is deliberate: if the sink wrote
back into the database Debezium watches, those writes would re-enter the binlog and create a
CDC feedback loop.

## Tech stack

| Concern        | Choice |
|----------------|--------|
| Source / sink DB | MySQL 8.4 LTS |
| CDC            | Debezium (Kafka Connect source) |
| Streaming      | Apache Kafka 4.x in **KRaft** mode (Confluent Platform 8.2.2, no ZooKeeper) |
| Stream processing | Kafka Streams, **Java 21**, Gradle |
| Serialization  | **Avro** + Confluent Schema Registry |
| Sink           | Kafka Connect JDBC sink (upsert) |
| Observability  | Kafka UI (kafbat), Prometheus + Grafana |
| Delivery guarantee | Debezium at-least-once + Streams `exactly_once_v2` + idempotent upsert ≈ effectively-once |

## Dataset

[Brazilian E-Commerce Public Dataset by Olist](https://www.kaggle.com/datasets/olistbr/brazilian-ecommerce)
— ~100k real (anonymized) orders across ~8 normalized tables, licensed
**CC BY-NC-SA 4.0** (non-commercial; fine for a portfolio). CDC is generated by replaying
each order's lifecycle timestamps as inserts and updates against the source database.

Download the nine CSVs into **`data/raw/`** (git-ignored). They must be present before the
**first** `docker compose up`, because the source DB loads them during initialization.

## Source database (step 2)

On first init, `mysql-source` runs the scripts in `mysql/source/initdb.d/`:

1. `01_schema.sql` — normalized Olist schema (9 tables, FKs on the order graph).
2. `02_load_dimensions.sql` — bulk-loads the reference tables (`customers`, `sellers`,
   `products`, category translation, `geolocation`) via server-side `LOAD DATA INFILE`
   from the CSVs bind-mounted read-only at `/csv`. The four **order lifecycle** tables
   (`orders`, `order_items`, `order_payments`, `order_reviews`) are left **empty** —
   the replayer (step 4) fills them over time so Debezium emits a realistic change stream.
3. `03_debezium_user.sql` — the least-privilege `debezium` user for the CDC connector.

To re-run init after changing these scripts, recreate the source with a fresh volume:

```bash
docker compose stop mysql-source && docker compose rm -f mysql-source
docker volume rm demodataengineeringproject2_mysql-source-data
docker compose up -d mysql-source
```

Verify the load:

```bash
docker exec mysql-source mysql -uroot -prootpw olist -e "SELECT COUNT(*) FROM customers;"   # 99441
docker exec mysql-source mysql -udebezium -pdbz -e "SHOW GRANTS FOR CURRENT_USER();"
```

## CDC connector (step 3)

The `connect` service is built from [connect/Dockerfile](connect/Dockerfile), which adds the
Debezium MySQL connector (3.5.2.Final) to the Confluent Connect image. Register the source:

```bash
curl -X POST -H "Content-Type: application/json" \
  --data @connect/debezium-source.json http://localhost:8083/connectors
```

Config highlights ([connect/debezium-source.json](connect/debezium-source.json)):
- `ExtractNewRecordState` unwrap SMT → topics carry the flat row state (materializable as
  `KTable`s); deletes become tombstones (`delete.tombstone.handling.mode=tombstone`).
- Avro key + value via Schema Registry; `decimal.handling.mode=double` to keep money fields
  simple downstream.
- `table.include.list` captures the 8 join tables (dimensions + lifecycle); `geolocation`
  is excluded.

Verify after registering:

```bash
curl -s http://localhost:8083/connectors/olist-mysql-source/status   # connector + task RUNNING
curl -s http://localhost:8081/subjects                               # mysql.olist.*-key/-value
docker exec kafka kafka-get-offsets --bootstrap-server localhost:9092 --topic mysql.olist.customers  # 99441
```

Topics for the four lifecycle tables (`orders`, `order_items`, `order_payments`,
`order_reviews`) appear only once the replayer (step 4) inserts rows.

## Replaying the order lifecycle (step 4)

[replayer/](replayer/README.md) is a Java 21 service that reads the order CSVs and replays
each order against `mysql-source` in **global timestamp order**: an INSERT at purchase time
(with its items + payment), an UPDATE per lifecycle timestamp (approved → shipped →
delivered), a final status correction for terminal states (e.g. canceled), and a review
INSERT at its creation time. Time is compressed so two years of history replay in minutes.

```bash
docker compose build replayer
docker compose run --rm replayer                              # full dataset
docker compose run --rm -e REPLAY_LIMIT_ORDERS=2000 replayer  # quick partial run
```

Tunables (env): `REPLAY_SPEEDUP` (real-ms ÷ this = sleep-ms; higher = faster),
`REPLAY_MAX_SLEEP_MS` (cap between events), `REPLAY_LIMIT_ORDERS` (0 = all),
`REPLAY_TRUNCATE_FIRST` (clears the lifecycle tables first, so it's re-runnable).

Because each order goes through several updates, the `orders` topic carries far more
records than there are orders — that's the near-real-time change stream:

```bash
docker exec kafka kafka-get-offsets --bootstrap-server localhost:9092 --topic mysql.olist.orders
```

## Denormalizer (step 5)

[streams-app/](streams-app/README.md) is a Java 21 Kafka Streams application that materializes
the eight CDC topics as `KTable`s and produces a flat `order_enriched` topic keyed by
`order_id`:

- orders ⨝ customers (KTable **foreign-key** join)
- per-order item aggregates (count, total price/freight, distinct sellers), where each item's
  product category is resolved via products ⨝ category-translation (a second FK join)
- per-order payment aggregates (count, total, types, max installments)
- per-order review aggregate (count, average score)

Runs with `processing.guarantee=exactly_once_v2`; values are Avro (Schema Registry), internal
state is JSON. It's behind the `app` profile because it needs the CDC topics to exist first.

```bash
docker compose up -d --build streams-app          # start it (after the connector is registered)
docker exec kafka kafka-get-offsets --bootstrap-server localhost:9092 --topic order_enriched
```

The output is the read model's change log — each order re-emits as its items, payments, and
reviews arrive, so the latest record per `order_id` is the current enriched view. Unit tests
(`TopologyTestDriver` + a mock Schema Registry) run during the image build (`gradle test`).

## Read-model sink (step 6)

A Debezium JDBC sink connector ([connect/jdbc-sink.json](connect/jdbc-sink.json)) consumes
`order_enriched` and **upserts** it into `mysql-sink` — closing the loop back into MySQL.
The connector + MySQL driver are baked into the Connect image ([connect/Dockerfile](connect/Dockerfile)).

```bash
curl -X POST -H "Content-Type: application/json" \
  --data @connect/jdbc-sink.json http://localhost:8083/connectors
```

Key settings: `insert.mode=upsert`, `primary.key.mode=record_value`, `primary.key.fields=order_id`,
`schema.evolution=basic` (auto-creates the table), `key.converter=StringConverter` (the topic
key is a plain string), `value.converter=AvroConverter`. It reads from `earliest` to backfill.

Inspect the read model:

```bash
docker exec mysql-sink mysql -uroot -prootpw olist_readmodel \
  -e "SELECT COUNT(*) FROM order_enriched; SELECT order_id, customer_city, item_count, total_price, product_categories, review_avg_score FROM order_enriched LIMIT 5;"
```

The full path: MySQL (source) → Debezium → Kafka → Kafka Streams → Kafka → JDBC sink →
MySQL (sink). One row per order, kept continuously in sync.

## Observability (step 7)

Optional Prometheus + Grafana stack ([observability/](observability/)), behind the
`observability` profile. `kafka-exporter` connects to Kafka as a client and exposes
consumer-group lag and per-topic offsets; Prometheus scrapes it; Grafana auto-provisions a
datasource and the "CDC denormalization pipeline" dashboard (lag + throughput).

```bash
docker compose --profile observability up -d
```

| Service | URL | Notes |
|---------|-----|-------|
| Grafana | http://localhost:3000 | anonymous viewer enabled; admin/admin to edit |
| Prometheus | http://localhost:9090 | |
| kafka-exporter | http://localhost:9308/metrics | |

The dashboard shows lag for the two pipeline consumer groups — `olist-denormalizer` (the
Streams app) and `connect-olist-readmodel-sink` (the JDBC sink) — plus topic throughput.

## Testing

- **Unit** — `DenormalizationTopologyTest` drives the topology with `TopologyTestDriver` and a
  mock Schema Registry (no broker). Runs in the Streams image build (`gradle test`).
- **Integration** — `PipelineIntegrationTest` (`src/integrationTest`) runs the real topology
  against an ephemeral Kafka broker via **Testcontainers**, producing Debezium-shaped Avro and
  asserting the enriched output over the wire. Run on a Docker-enabled host:
  ```bash
  cd streams-app && gradle integrationTest
  ```
  (Note: Docker Desktop's in-container socket is a proxy Testcontainers can't drive, so this
  runs from the host or Linux CI, not docker-in-docker.)
- **Live end-to-end smoke** — with the stack running, insert an order into `mysql-source` and
  watch it reach `mysql-sink`:
  ```bash
  docker exec mysql-source mysql -uroot -prootpw olist -e \
    "INSERT INTO orders(order_id,customer_id,order_status,order_purchase_timestamp) \
     SELECT 'smoketest0000000000000000000001', customer_id, 'delivered', NOW() FROM customers LIMIT 1;"
  docker exec mysql-sink mysql -uroot -prootpw olist_readmodel -e \
    "SELECT order_id, customer_city, order_status FROM order_enriched WHERE order_id='smoketest0000000000000000000001';"
  ```

## Running step 1

Requires Docker Desktop (with Compose v2).

```bash
docker compose up -d        # start everything
docker compose ps           # all services should report (healthy)
docker compose down         # stop; add -v to also wipe the data volumes
```

### Endpoints

| Service | URL / port | Notes |
|---------|-----------|-------|
| Kafka UI | http://localhost:8088 | topics, messages, connectors, lag |
| Schema Registry | http://localhost:8081 | `GET /subjects` |
